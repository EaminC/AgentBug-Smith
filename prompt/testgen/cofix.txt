### CRITICAL: Diagnosing the F2P Feedback
A successful F2P run requires the test to FAIL before the patch (catching the bug) and PASS after the patch. 
Look at the exit codes in the feedback (`rc1` = before patch, `rc2` = after patch) to diagnose the issue:

1. **Test is Skipping (Exit Code 5):** If the test says "skipped", YOU MUST REMOVE all defensive `try...except` blocks and `skip()` logic from the test file. Let imports crash naturally so the pipeline can heal them.
2. **Missing Module / ImportError:** You MUST add `RUN pip install -e .` (or `{language}` equivalent like `npm install`) to the END of the Dockerfile. The local repository itself must be installed into the environment.
3. **File Not Found Error:** The Dockerfile MUST contain `COPY . .` to ensure the dynamically injected test file is mounted into the container. DO NOT use partial copies like `COPY src ./src`.
4. **False Positive (rc1=0, rc2=0):** If both runs pass, the test is NOT capturing the bug. You must rewrite the test logic to enforce stricter assertions that fail on the buggy implementation.
5. **False Negative (rc1=1, rc2=1):** If both runs fail, either the environment is broken, or the test is asserting something the patch doesn't actually fix.

### CRITICAL: Environment Robustness (Self-Healing)
1. **Metadata Detection:** Use shell logic (`if [ -f file ]; then ...`) to check for configuration files before executing install commands.
2. **Dynamic Key Retrieval:** Model initializations in the test script MUST use environment variables (e.g., `os.getenv('OPENAI_API_KEY')`).
3. **Editable Installation (CRITICAL):** You MUST install the local project into the environment using the standard command for {language}.
  * For Python: Use `RUN pip install -e .` unconditionally. Do NOT wrap it in `if [ -f "setup.py" ]`.
  * For Node.js/TypeScript: Use `RUN npm install` (or yarn/pnpm equivalent).
  * For Rust: Use `RUN cargo build`.
4. **Framework Async Patterns:** If the {language} agent framework utilizes asynchronous coroutines, ensure they are explicitly awaited.

### CRITICAL: Mocking & API Calls
1. **Mocking Network Calls:** If the feedback shows an `APIConnectionError`, `BadRequestError`, `Timeout`, or any HTTP 400/401 error, it means your test attempted a real network request with dummy data. You MUST intercept the API response and return a valid mock object using the appropriate tool for {language}:
   * For Python: Use `@patch`, `patch.object`, or `AsyncMock` from `unittest.mock`.
   * For Node.js/TypeScript: Use `jest.mock()`, `vi.mock()`, or `sinon` depending on the detected testing framework.
2. **Do NOT Mock the Test Subject:** If your F2P test is failing because assertions aren't triggering, check your mocks. If you mock the exact function where the bug lives, the test will never execute the patch. Only mock the underlying network/API client, not the wrapper logic.
3. **Do NOT Simulate the Bug:** Do not write fake implementations of the buggy or fixed code to force an error. The pipeline physically swaps the underlying files on disk using `git apply` between Phase B and Phase E. Your only job is to test the real imported code.
